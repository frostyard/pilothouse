# Fleet management implementation plan

## Status

This document is a proposed implementation plan. The fleet pages currently contain preview data and disabled controls. No pairing, network transport, remote query, or remote action described here exists yet.

The plan deliberately requires every participating machine to share a routed administrative network. Supported deployments include a private LAN, a routed management VLAN, WireGuard, Tailscale, or another VPN that provides direct controller-to-target reachability. Pilothouse will not provide NAT traversal, port forwarding, an internet relay, or store-and-forward delivery.

## Product decisions

The first release will use the following constraints:

- One Pilothouse installation acts as the controller for many targets.
- A target may be paired with one controller.
- The controller can directly route to every target over an administrative network.
- A target administrator creates the invitation that permits pairing.
- Pairing requires authenticated administrators on both the controller and target.
- Version 1 uses the existing controller-side administrator decision for all fleet capabilities; per-human capability policy is deferred.
- The invitation is a short-lived bootstrap secret, not a standing credential.
- Long-term trust uses pinned machine identities, mutual TLS, and explicit target-side capability grants.
- Read-only inventory ships before any remote mutation.
- Targets do not queue actions while offline.
- Local Pilothouse management remains available when the controller or network is unavailable.
- The target broker remains the final authority for authorization, validation, confirmation, locking, execution, and audit.

## Goals

- Show current and cached state for paired Pilothouse systems from one controller UI.
- Pair systems without requiring a shared public certificate authority.
- Keep all network-facing parsing and TLS handling outside the root broker.
- Preserve fixed broker query and action IDs with exact parameter validation.
- Let each target grant and revoke specific remote capabilities independently.
- Attribute each remote mutation to both the authenticated controller and the initiating human operator.
- Prevent replay and unsafe automatic retries of remote mutations.
- Make stale state, offline targets, incompatible versions, and uncertain action outcomes explicit in the UI.
- Keep the protocol bounded, versioned, testable, and suitable for LAN or VPN use.

## Non-goals

- Public internet discovery or connectivity without a routed VPN.
- NAT traversal, UPnP, relay infrastructure, or hosted coordination.
- Multiple active controllers for one target.
- Controller high availability or distributed fleet locks.
- Offline mutation queues or scheduled delivery when a target reconnects.
- Remote shell access, arbitrary command execution, or terminal streaming.
- Generic filesystem access, generic HTTP forwarding, or socket proxying.
- Dynamic registration of methods advertised by a peer.
- Wildcard capability grants such as `actions:*`.
- Automatic trust after a reinstall, identity change, or cloned-machine detection.

## Terminology

| Term | Meaning |
| --- | --- |
| Controller | The Pilothouse installation providing the combined fleet UI. |
| Target | A Pilothouse installation managed from the controller. |
| Machine identity | A random stable ID and signing key generated by a broker. |
| Link | The unprivileged process that terminates fleet TLS and transports bounded messages. |
| Invitation | A short-lived, single-use target-generated pairing secret. |
| Peer | A controller or target whose machine identity has been explicitly approved and pinned. |
| Grant | The target-side list of capabilities delegated to one controller. |
| Capability | A compiled, versioned query or action identifier with a fixed schema. |

## Required architecture

```text
browser
  |
  v
controller pilothouse
  |
  | local fixed broker query or action
  v
controller pilothoused
  |  authenticate current operator
  |  re-resolve local groups
  |  enforce controller policy
  |  record intent
  |  sign remote envelope
  v
controller pilothouse-link
  |
  | direct HTTPS with mutual TLS over LAN or VPN
  v
target pilothouse-link
  |
  | dedicated local fleet socket
  v
target pilothoused
     verify machine signature and target binding
     enforce target grant and replay protection
     derive canonical resource from live state
     validate confirmation and acquire local lock
     record target audit intent
     execute an existing fixed query or action
```

The existing browser-facing `pilothouse` process and root `pilothoused` broker retain their current boundary. The existing `/run/pilothouse/broker.sock` protocol must never be exposed or transparently forwarded over the network.

On the controller, link requests and target responses travel over the dedicated local fleet Unix socket as one bounded request/response exchange. The controller broker never dials the target or reads a network response directly.

### New link process

Add a `pilothouse-link` command with these responsibilities:

- Listen only on an explicitly configured administrative address and port.
- Initiate direct HTTPS requests from a controller to targets.
- Terminate TLS 1.3 and enforce pinned peer transport keys.
- Apply connection, header, body, and response timeouts.
- Enforce maximum request and response sizes before forwarding.
- Pass complete signed envelopes to a dedicated local fleet broker socket.
- Return broker-produced signed responses without changing their contents.
- Emit connection and protocol metrics that contain no secrets.

The link must not:

- Open the existing browser broker socket as a generic client.
- Forward arbitrary methods, URLs, paths, headers, or byte streams.
- Hold the broker's machine signing key.
- Decide whether an operator or controller may execute an action.
- Decode a privileged payload and synthesize a different broker operation.
- Follow redirects supplied by a target or controller.

### Dedicated local fleet socket

Add a second root-owned Unix socket, for example `/run/pilothouse/fleet.sock`, accessible only to the `pilothouse-link` service account. Its protocol should contain a small fixed set of operations for pairing, signed-envelope verification, inventory production, remote dispatch, and operation-result lookup.

The socket is a narrower interface than the browser broker socket. It must not expose PAM login, browser sessions, arbitrary local query dispatch, or arbitrary local action dispatch.

The target fleet socket maps a verified envelope through a separate compiled fleet capability table. It must never use an envelope capability string as an unrestricted key into the existing `ActionRegistry` or `QueryRegistry`. A valid local broker action that has not been explicitly mapped as a fleet capability remains unreachable through this socket.

## Security invariants

The implementation is not acceptable unless all of these remain true:

1. `pilothoused` has no TCP listener and performs no network dialing.
2. Network-facing code runs without root privileges.
3. The browser session token, password, and CSRF token never leave the controller host.
4. Pairing invitations are short-lived, single-use, and separate from browser sessions and machine credentials.
5. Display names, hostnames, DNS names, and IP addresses are never authoritative machine identities.
6. A target authorizes the pinned controller machine and explicit granted capabilities, not a controller-asserted local account.
7. The target treats the human actor as an issuer-qualified audit claim, not as a target-local UID.
8. The target recomputes resources and confirmation values from live local state.
9. Local and remote mutations use the same target-side resource locks and domain validation.
10. Audit intent is durably committed before mutation; audit failure prevents execution.
11. A duplicate operation ID cannot execute a mutation twice.
12. A timeout or disconnect after submission is an uncertain outcome, not permission to retry automatically.
13. Capability negotiation is the intersection of locally compiled identifiers; peers cannot create methods dynamically.
14. One target's credential cannot authenticate to another target.
15. Revocation is enforceable locally by the target without contacting the controller.

### Residual controller risk

A target authorizes the controller machine, not the controller's individual human accounts. The actor attestation identifies who the controller claims initiated an operation, but a compromised controller can forge that claim and exercise every capability in its active target grant. Version 1 contains this risk through narrow grants, short mutation-grant expiration, target-local revocation, and independent target audit; it does not cryptographically prove the human identity to the target.

## Machine identity

Each broker generates identity on first fleet enablement:

- A cryptographically random 128-bit or larger machine ID.
- An Ed25519 signing key pair.
- A key generation number beginning at one.
- Creation and rotation timestamps.

The private signing key belongs to `pilothoused` and is stored under `/var/lib/pilothouse` with mode `0600`. It is used to sign protocol payloads, grants, transport-key rotations, and actor attestations. The unprivileged link process receives signed bytes but not the private key.

The link maintains a separate TLS transport key. Its certificate is self-issued or broker-attested and is pinned during pairing. Transport keys may rotate without changing machine identity when the new key is authorized by a machine-identity signature.

Routine transport-key rotation requires an overlap in which the old pinned transport key remains valid long enough to carry the machine-signed new-key authorization. If the old key is unavailable or suspected compromised, in-band rotation is not allowed and the systems must pair again.

Hostname, Tailscale name, IP address, operating-system image, and role are mutable metadata associated with a machine ID.

### Cloning, restore, and reinstall

- Fleet identity must not be included in a golden image.
- Packaging must generate identity after first boot or first fleet enablement.
- Two simultaneous connections using one machine identity cause quarantine and an operator-visible alert.
- A restored disk snapshot may contain stale grants and replay state; documented recovery requires identity reset and re-pairing unless continuity can be proven.
- A reinstalled host with the same name or address but a different key is a new machine and requires explicit pairing.
- Provide a local administrative identity-reset command that revokes local grants before deleting identity material.

## Pairing design

### Invitation contents

The target creates a copyable or QR-encodable value containing:

```text
format version
target endpoint
target machine ID
target machine signing public key
target TLS public-key fingerprint
invitation ID
256-bit random secret
expiration timestamp
initial capability bundle identifier
```

Use a recognizable prefix such as `ph1_` followed by base64url-encoded structured data. The value must not be placed in a URL query string, command argument that may be logged, audit record, or HTML response after initial display.

The target stores only a verifier derived from the random secret. The full invitation is displayed once. Expiration should default to ten minutes and never exceed one hour.

### Pairing state machine

```text
created
  | controller proves invitation secret over target-pinned TLS
  v
reserved
  | target records proposed controller key and requested capabilities
  v
pending-target-approval
  | authenticated target administrator approves exact peer and grant
  v
active

created/reserved/pending-target-approval
  -> expired
  -> cancelled

active
  -> revoked
  -> incompatible
```

An invitation reserved by one controller key cannot be completed by another key. Failed proof attempts are rate-limited and counted. Reaching the failure limit cancels the invitation.

### Pairing sequence

1. A target administrator requests an invitation through a fixed local broker action.
2. The target broker creates and durably records the invitation verifier and expiry.
3. The UI displays the invitation once and warns that it grants a pairing attempt.
4. A controller administrator pastes the invitation into the controller fleet page.
5. The controller validates the format, expiry, endpoint, target ID, and embedded key fingerprints locally.
6. The controller link connects directly to the target endpoint without following redirects.
7. The controller pins the target TLS key from the invitation before sending the secret.
8. The controller broker supplies its machine identity, transport key, requested capabilities, and a signed pairing statement.
9. The target verifies the secret and signature, reserves the invitation to that controller key, and records a pending peer.
10. The target UI shows the controller fingerprint, endpoint metadata, and exact requested capabilities.
11. An authenticated target administrator approves or rejects the request.
12. Approval creates a target-side grant and signed response.
13. The controller verifies the target response and stores the pinned target identity, transport key, and accepted grant.
14. The target atomically consumes the invitation.
15. All later requests require mutual TLS, pinned keys, signed payloads, and an active grant.

Pairing is incomplete until both sides have durable matching peer records. Interrupted pairing may resume only with the same invitation ID and controller key while the invitation remains valid.

## Capability and authorization model

Capabilities are fixed IDs with semantic revisions, for example:

```text
org.frostyard.pilothouse.system.summary@1
org.frostyard.pilothouse.attention.summary@1
org.frostyard.pilothouse.maintenance.state@1
org.frostyard.pilothouse.services.state@1
org.frostyard.pilothouse.services.restart@1
org.frostyard.pilothouse.sysext.update@1
org.frostyard.pilothouse.maintenance.reboot@1
```

A target grant records:

- Grant ID.
- Controller and target machine IDs.
- Monotonically increasing grant epoch.
- Exact query and action capability IDs and revisions.
- Optional resource restrictions such as an allowlist of service units.
- Creation, expiration, and revocation timestamps.
- Target-local administrator who approved the grant.

The target includes a signed summary of the active grant ID, epoch, expiration, and capability digest in its hello response. A controller that presents a stale epoch refreshes this summary and displays `grant-changed`, `grant-expired`, or `revoked` rather than retrying requests indefinitely.

UI bundles such as Monitoring or Operations may be offered, but a bundle is expanded into exact capabilities when the grant is written. The target never stores a wildcard.

A remote operation is authorized only when:

1. The controller has freshly resolved the initiating human's local NSS identity and groups.
2. In version 1, the human is a current controller administrator and the requested operation has a compiled controller-side fleet mapping. Per-human capability policy is future work.
3. The target verifies the controller machine signature and target binding.
4. The target grant is active and its epoch matches the request.
5. The grant contains the exact capability and semantic revision.
6. The target has a compiled implementation for that capability and revision.
7. The target manager validates all parameters and rediscovers the resource from live state.

Do not create `auth.Identity{Admin: true}` for a controller. Introduce an explicit actor/principal model that distinguishes locally verified users from remote controller principals. The actor claim supplied by a controller is for attribution; the target grant authorizes the controller machine.

## Protocol

### Encoding and signatures

The initial implementation may retain bounded JSON to minimize dependencies. Signatures are calculated over the exact raw payload bytes produced by the broker, not over a reserialized JSON value. The link forwards those bytes unchanged. The receiving broker verifies the signature before strict decoding with unknown fields rejected.

A transport envelope contains only bounded routing metadata, raw signed payload bytes, key generation, and signature. Sender ID, target ID, request ID, protocol version, capability, and expiry are repeated inside the signed payload and verified after decoding.

If the protocol later adopts Protobuf or another encoding, it must specify deterministic serialization before signatures are migrated.

### Version negotiation

The signed hello response includes:

- Machine ID and key generation.
- Protocol major and supported minor range.
- Build version.
- Maximum request and response sizes.
- Supported capability IDs and semantic revisions.
- Transport and signing-key fingerprints.
- Random challenge nonce.
- Active grant ID, epoch, expiration, and capability digest when paired.

Different protocol majors are incompatible. Minor features are selected by intersection. There is no fallback to unsigned requests, bearer-only authentication, or an older pairing protocol below the configured security floor.

### Network endpoints

The link may expose a small fixed surface such as:

```text
GET  /v1/fleet/hello
POST /v1/fleet/pair
POST /v1/fleet/query
POST /v1/fleet/action
POST /v1/fleet/operations/status
```

These endpoints carry typed fleet envelopes. They are not generic broker endpoints. Each endpoint has an independent body limit, timeout, rate limit, and allowed message type.

### Request requirements

Every authenticated query or action payload includes:

- Protocol major and minor.
- Sender and target machine IDs.
- Sender key generation.
- Grant ID and epoch.
- Random request or operation ID.
- Capability ID and revision.
- Exact typed parameters.
- Creation and expiration times.
- TLS connection challenge or session nonce.
- Actor attestation when initiated by a human.
- Signature over all preceding fields.

The hello challenge establishes handshake freshness. A TLS-session nonce binds each request to that negotiated connection. A durable operation ID deduplicates mutations across connections and restarts. Read-only queries use request IDs and session binding but do not enter the durable mutation ledger. Wall-clock expiry limits exposure but is not the only replay defense. Implement and test a bounded clock-skew allowance.

### Error model

Return stable categories rather than backend error strings:

- unauthenticated
- peer-revoked
- grant-expired
- capability-denied
- incompatible-version
- invalid-parameters
- resource-not-found
- resource-conflict
- confirmation-required
- replayed-operation
- rate-limited
- unavailable
- timeout
- unknown-outcome

Detailed backend errors remain in protected local logs when appropriate and are not transported to the controller by default.

## Read-only inventory

The controller polls each active target directly. Recommended defaults are a lightweight hello every 30 seconds and a summary inventory every two minutes, with jitter. Detailed views perform bounded on-demand queries.

An inventory snapshot includes:

- Target machine ID and display metadata.
- Snapshot schema revision.
- Observation timestamp and freshness duration.
- Image, kernel, uptime, and reboot posture.
- Attention counts by severity.
- Update summary.
- Service and workload summary counts.
- Target signature.

The controller stores the last accepted snapshot and displays one of these states:

- current: recent authenticated snapshot
- stale: snapshot exceeded its freshness duration
- offline: target cannot be reached and cached data exists
- never-seen: paired but no valid snapshot has been accepted
- revoked: local or target grant is revoked
- incompatible: protocol or required capability revisions do not intersect
- grant-changed: target grant metadata differs from the controller's cached grant
- identity-conflict: the same machine identity appeared from conflicting peers

The UI always displays `observed_at` and `last_seen`. Cached information must never look live.

Logs, journals, environment variables, secrets, and unbounded command output are excluded from summary inventory. Journal and container-log access require separate reviewed capabilities with existing line, time, and byte limits.

## Remote query flow

1. The browser requests a fixed fleet page or detail view.
2. The controller web module calls a fixed local broker query.
3. The controller broker re-resolves the human identity and validates the target machine ID.
4. A compiled remote-query definition selects one exact target capability and schema.
5. The controller broker signs a bounded query payload.
6. The controller link sends it to the target's persisted endpoint over pinned mTLS.
7. The target link forwards the unchanged envelope to the target fleet socket.
8. The target broker verifies identity, grant, revision, expiry, nonce, and parameters.
9. The target invokes one fixed local query implementation.
10. The target broker signs the bounded result.
11. The controller verifies and returns a narrow presentation model to the web module.

Before exposing queries remotely, evolve `QueryRegistry` toward action-like exact parameter definitions and response limits. This is a migration of every existing query registration and local caller, not a fleet-only wrapper. Remote query code must not use an arbitrary capability ID supplied by the browser.

## Remote action flow

### Controller processing

1. Before showing a destructive confirmation form, the controller invokes the compiled preview method for that exact fleet action.
2. The target derives the canonical resource from live state and returns a signed, short-lived preview ID and confirmation value.
3. The controller broker verifies the preview signature and the UI displays that target-derived value.
4. The browser submits POST with the existing CSRF and origin protections, the preview ID, and exact confirmation.
5. The fleet handler calls `ValidateAction` and compares against the verified preview value.
6. The controller broker re-resolves the current human identity and authorizes the compiled fleet action.
7. The controller validates the paired target and cached grant, but does not assume the target will accept it.
8. The controller creates a random 256-bit operation ID.
9. Controller audit intent is written before dispatch.
10. The controller constructs and signs the exact target action envelope, including the preview ID and confirmation.
11. The controller link sends the operation once.

### Target processing

1. Verify mutual TLS pin, machine signature, sender, target, protocol version, expiry, connection nonce, grant, epoch, capability, and exact parameters.
2. Atomically insert `(controller machine ID, operation ID)` into the durable replay ledger.
3. Return the existing stored state without executing if the same operation was previously accepted.
4. Recompute the canonical target resource from live target state.
5. Verify the preview ID is unexpired and bound to the same controller, target, action, and resource.
6. Compare destructive confirmation with the newly target-derived canonical resource and reject if state changed since preview.
7. Acquire the same target-local resource lock used by local actions.
8. Write target audit intent before mutation.
9. Execute the existing fixed manager operation.
10. Durably record success, failure, or unknown outcome.
11. Sign and return the stored operation state.

### Operation states

```text
created -> submitted -> accepted -> running -> succeeded
                                     |       -> failed
                                     |       -> unknown
                                     -> rejected
```

The controller may query the same operation ID after a timeout. It must not generate a replacement operation automatically. A target crash after the underlying mutation but before result persistence produces `unknown`, even when the controller can infer a likely outcome later.

Reboot and other connection-breaking actions acknowledge durable acceptance before execution. Reconnection may provide evidence of completion, but the audit record retains any period of uncertainty.

## Audit and attribution

Controller and target both record remote mutations.

Controller records contain:

- Controller-local username and UID.
- Controller and target machine IDs.
- Grant and operation IDs.
- Fixed capability and canonical target resource.
- Submission, acceptance, and observed completion times.
- Stable outcome or uncertainty category.

Target records contain:

- Verified controller machine ID and key generation.
- Actor username and UID as controller-issued claims.
- Grant and operation IDs.
- Fixed local action and target-derived resource.
- Acceptance, start, completion, and stable outcome.

Extend the audit actor model to identify local users separately from remote issuer-qualified actors. A remote UID is never treated as a target-local UID. Existing local audit records remain valid through optional versioned fields or a schema migration.

Inventory polling and routine heartbeats should produce metrics and bounded operational logs, not privileged action audit records. Pairing, grant changes, revocation, identity reset, and every remote mutation are audited.

## Configuration and packaging

Add a dedicated system account and unit for `pilothouse-link` with systemd hardening equivalent to the web process. The service requires network access and access to the dedicated fleet socket, but not the browser broker socket or root-equivalent engine sockets.

Possible configuration:

```text
PILOTHOUSE_FLEET_LISTEN=100.80.10.4:9443
PILOTHOUSE_FLEET_PUBLIC_ENDPOINT=https://host.tailnet.ts.net:9443
PILOTHOUSE_FLEET_SOCKET=/run/pilothouse/fleet.sock
PILOTHOUSE_FLEET_CONNECT_TIMEOUT=5s
PILOTHOUSE_FLEET_REQUEST_TIMEOUT=30s
```

The listen address has no public wildcard default. Fleet networking remains disabled until an administrator explicitly configures it. The advertised endpoint is a connection hint and must match the identity pin from pairing.

Firewall and VPN policy remain deployment responsibilities. Documentation should include examples for a private LAN, WireGuard, and Tailscale, without automating firewall or VPN changes.

## UI plan

### Controller fleet list

Replace static records with:

- Pinned machine identity and editable display name.
- Endpoint and transport status.
- Current, stale, offline, revoked, incompatible, or conflict state.
- Last seen and last observed timestamps.
- Image, kernel, update, reboot, and attention summaries.
- Link to the system detail page.
- Local action to forget a revoked peer.

### Target pairing page

Provide:

- Fleet networking disabled/configured state.
- Create invitation action with expiration and initial read-only bundle.
- One-time invitation display and explicit secrecy warning.
- Pending controller fingerprint and requested capability review.
- Approve, reject, revoke, rotate, and reset-identity actions.
- History of pairing and grant changes.

### Controller enrollment page

Provide:

- Paste or scan invitation input.
- Parsed target endpoint, machine ID, and fingerprint preview.
- Requested capability selection from compiled bundles.
- Pairing progress with stable failure categories.
- Explicit confirmation before storing the paired target.

### Target detail page

Deliver sections incrementally:

- Read-only identity, freshness, image, kernel, uptime, and attention.
- Resource and service summaries.
- Granted capability display.
- On-demand bounded details.
- Remote action controls only when target and local policy both permit them.
- Operation state and dual-audit correlation by operation ID.

All pages continue to work without JavaScript. Pairing and mutations use POST, CSRF validation, origin checks, 303 redirects, and broker authorization. HTMX remains progressive enhancement.

## Delivery phases

### Phase 0: Architecture decision and protocol specification

Outcome: approved design with no production network surface.

Work:

- Record the routed LAN/VPN requirement and one-controller constraint.
- Define machine identity, transport key, grant, actor, and operation semantics.
- Specify invitation and peer state machines.
- Specify exact message schemas, limits, signatures, and stable errors.
- Choose initial read-only capabilities and first mutation.
- Document key storage, backup, restore, cloning, and reinstallation behavior.
- Threat-model controller compromise, target compromise, link compromise, token theft, replay, downgrade, and stale state.
- Prototype Go TLS pinning and raw-payload signing in an isolated test package.

Automated tests:

- Test vectors for signatures, key pins, token parsing, and version negotiation.
- Measured maximum token and message sizes.

Manual review gates:

- Independent security review of the protocol document.
- Review confirms the fleet socket has no path that transparently proxies the existing broker socket.
- Negative-test design covers valid local broker action IDs that are absent from the fleet allowlist.

Exit criteria:

- Protocol major version 1 is fully specified.
- Security invariants and non-goals are accepted.
- Every trust-boundary decision in the open-decision list is resolved and recorded.

### Phase 1: Broker safety foundations

Outcome: local-only code can represent remote actors, strict queries, grants, and operations without networking.

Work:

- Introduce an actor type distinguishing local identity from remote controller claims.
- Preserve current NSS re-resolution and `Admin` behavior for local operations.
- Extend audit records with optional issuer, machine, grant, and operation fields.
- Add target-qualified canonical resource helpers.
- Evolve every existing query registration and local caller to exact parameter names and response-size policy, with compatibility tests for all current queries.
- Add a compiled remote capability mapping; do not accept browser-supplied action IDs.
- Add fleet bbolt schema and transactional helpers.
- Add durable operation insertion, lookup, transition, and restart recovery.
- Add grant epoch and capability intersection logic.

Tests and review gates:

- Existing local broker and module tests remain unchanged in behavior.
- Unexpected query parameters are rejected.
- Old audit records remain readable.
- Audit-store failure blocks simulated remote mutation.
- Duplicate operation insertion cannot execute twice under concurrency.
- Restart converts accepted/running simulated operations and their open audit records to unknown.
- A signed envelope naming a real local broker action that is absent from the fleet capability table is rejected.

Exit criteria:

- All foundations are exercised without a TCP listener.
- Local action safety, confirmation, and locking tests still pass.
- Data migrations and file permissions are verified.

### Phase 2: Machine identity and local lifecycle

Outcome: each broker has durable identity and can manage peers and grants locally.

Work:

- Generate Ed25519 machine identity on explicit fleet enablement.
- Store private material with root-only permissions.
- Add identity query and administrative reset/rotation actions.
- Implement peer and grant CRUD behind fixed broker queries/actions.
- Implement revocation and monotonically increasing grant epochs.
- Add duplicate machine-identity state and quarantine handling.
- Add CLI diagnostics that print only public identity and fingerprints.

Tests and review gates:

- Permissions and ownership are checked after create, restart, and upgrade.
- Private keys never appear in logs, audit, query results, or errors.
- Identity persists across normal restart.
- Reset requires exact destructive confirmation.
- Revoked grants reject all later simulated requests.
- Key rotation verifies old/new signatures and rejects unauthorized replacement.

Exit criteria:

- Identity and grants are production-safe without networking enabled.
- Backup and clone behavior is documented and tested where automatable.

### Phase 3: Link service and pairing

Outcome: two directly routed systems can pair securely, but no remote inventory or action is enabled.

Work:

- Add `pilothouse-link` and hardened systemd packaging.
- Add the dedicated fleet Unix socket and fixed local link protocol.
- Implement explicit listen and public-endpoint configuration.
- Implement TLS 1.3, transport-key creation, and certificate pinning.
- Implement target invitation creation, expiration, reservation, approval, consumption, and cancellation.
- Implement controller invitation parsing and endpoint validation.
- Implement controller and target peer approval views.
- Implement signed hello and protocol negotiation.
- Return target responses to the controller broker only through the same bounded fleet Unix socket request/response exchange.
- Add per-source and per-invitation rate limits.
- Refuse redirects and key changes outside the approved rotation flow.

Tests and review gates:

- Invitation is rejected after expiry, cancellation, consumption, or excessive failures.
- Invitation reserved to one controller key rejects another key.
- A stolen secret without the pinned target key cannot complete pairing through a substituted endpoint.
- TLS key substitution and protocol downgrade are rejected.
- Unknown JSON fields, oversized bodies, slow requests, and malformed certificates are rejected.
- Pairing interruption resumes only with matching keys and an unexpired invitation.
- Fuzz invitation and pairing parsers.

Exit criteria:

- Pairing succeeds over a LAN and Tailscale test network.
- Both sides persist matching pins and grants.
- Packet-capture inspection finds neither the raw invitation secret, the `ph1_` token, nor private key material after pairing completes.

### Phase 4: Read-only fleet MVP

Outcome: the controller fleet page shows authenticated target summaries and honest freshness.

Version 1 restricts the controller fleet list, cached snapshots, and target detail pages to current controller administrators.

Initial capabilities:

```text
system.summary@1
attention.summary@1
maintenance.state@1
```

Work:

- Implement signed summary inventory production at the target broker.
- Implement direct controller polling with jitter and bounded concurrency.
- Verify target identity and signature before accepting a snapshot.
- Cache only the latest bounded snapshot per target.
- Replace static fleet preview data with fixed broker query results.
- Implement current, stale, offline, never-seen, revoked, incompatible, and conflict states.
- Add target detail identity and summary sections.
- Add per-target and fleet-wide request budgets.
- Expose last-seen, last-observed, latency, and stable connection errors.

Tests and review gates:

- A snapshot signed by the wrong target is rejected.
- Endpoint changes do not change target identity.
- Stale and offline data are visibly distinct from current data.
- One slow or malicious target cannot block other target refreshes.
- Snapshot and fleet cardinality limits prevent memory or database growth without bounds.
- Read-only operation works when the target web UI is stopped but broker/link remain available.

Exit criteria:

- Read-only fleet management is useful without enabling any remote mutation.
- Local management continues during controller outage.
- UI and protocol behavior are documented for LAN, WireGuard, and Tailscale.

### Phase 5: On-demand remote queries

Outcome: authenticated controller administrators can request bounded detail from one target.

Candidate capabilities:

```text
services.state@1
containers.summary@1
jobs.state@1
activity.summary@1
```

Work:

- Add compiled controller-to-target query mappings.
- Enforce refreshed controller-side human authorization.
- Enforce target grant and exact query schema.
- Add bounded query timeouts and cancellation.
- Add short-lived controller cache only where semantics are explicit.
- Display observation timestamps on every detailed result.
- Review journal and container-log access separately before implementation.

Tests and review gates:

- Browser input cannot choose an arbitrary remote query ID.
- Extra, missing, oversized, and malformed parameters are rejected on both brokers.
- Target authorization cannot be bypassed by controller UI state.
- Revocation during an in-flight query prevents new queries.
- Results exceeding configured limits are rejected rather than truncated ambiguously.

Exit criteria:

- Each query has a documented sensitivity, parameter schema, result bound, timeout, and grant requirement.
- No root-equivalent socket or generic output crosses the network.

### Phase 6: First remote mutation

Outcome: one carefully selected action proves the complete remote operation model.

Use service restart as the first action. Restrict the initial grant to an explicit allowlist of service units. Do not begin with reboot, image deletion, or a broad container action.

Work:

- Add one fixed controller fleet action mapped to one fixed target action.
- Add a fixed signed action-preview exchange that obtains the target-derived canonical confirmation value before rendering the form.
- Bind preview IDs to controller, target, action, resource, and a short expiration.
- Implement actor attestation, controller audit, and signed operation envelope.
- Implement target replay insertion before execution.
- Reuse target-side resource derivation, manager rediscovery, confirmation, lock, and audit.
- Implement operation status lookup by the same operation ID.
- Display accepted, running, succeeded, failed, rejected, and unknown states.
- Ensure network timeout never creates an automatic replacement operation.

Tests and review gates:

- Replayed envelopes return stored state without a second restart.
- Expired, substituted, or state-stale preview IDs are rejected before mutation.
- Controller and target disagreeing about canonical resource causes rejection.
- Concurrent local and remote operations serialize through the same target lock.
- Controller group removal takes effect before the next remote action.
- Grant revocation prevents new actions immediately.
- Link compromise simulation cannot forge a broker signature.
- Target audit failure blocks execution even after controller audit succeeds.
- Disconnect before response produces status lookup, not retry.

Exit criteria:

- Dual audit records correlate by operation ID.
- At-most-once acceptance is demonstrated under retries, concurrency, restart, and packet loss.
- Manual security review approves adding further mutations.

### Phase 7: Expanded operations

Outcome: additional fixed actions are added one by one using the proven model.

Suggested order:

1. Explicitly allowlisted service start and stop.
2. Extension refresh and update as durable background operations.
3. Container restart with full target-discovered IDs.
4. Reboot with exact target-qualified confirmation.
5. Destructive container or image removal only after separate review.

Each added action requires:

- A unique controller capability mapping and semantic revision.
- Exact parameters and target-side rediscovery.
- Documented grant scope and confirmation behavior.
- Stable error categories and timeout semantics.
- Replay, lock, revocation, dual-audit, and uncertain-outcome tests.
- UI tests proving disabled or absent controls when capability is not granted.

Exit criteria:

- No action is enabled by a broad transport or module wildcard.
- Every remote action has equivalent or stronger safety than its local form.

### Phase 8: Lifecycle and production hardening

Outcome: fleet operation is supportable through normal upgrades and failure scenarios.

Work:

- Complete machine and transport key rotation, including old-key overlap for routine transport rotation and mandatory re-pairing when the old key is unavailable.
- Add grant expiration warnings and renewal workflow.
- Add explicit decommission, revoke, forget, and re-pair flows.
- Add duplicate-identity quarantine recovery.
- Exercise clock skew and certificate validity failures.
- Add bounded metrics for pairing, request rates, latency, stale targets, revocation, and operation outcomes.
- Add protocol fuzzing and long-running malformed-peer tests.
- Load-test expected fleet size with slow, offline, and hostile targets.
- Test mixed-version rolling upgrades and hard protocol-major rejection.
- Verify startup reconciliation keeps remote operation and audit unknown outcomes consistent after connection-breaking actions such as reboot.
- Document backup, restore, clone, reinstall, controller loss, and target replacement procedures.
- Add packaging upgrade tests for users, groups, sockets, state directories, and systemd hardening.

Exit criteria:

- Operators can rotate, revoke, replace, and recover systems without editing the database.
- Resource usage remains bounded at the documented fleet size.
- Security and operations runbooks are complete.

## Cross-phase test matrix

| Area | Required coverage |
| --- | --- |
| Pairing | Expiry, single use, reservation, cancellation, wrong key, stolen secret, interruption, rate limit. |
| Authentication | mTLS pinning, machine signatures, key generation, target binding, downgrade rejection. |
| Authorization | Current controller user, target grant, epoch, exact capability, resource restriction, revocation. |
| Replay | Duplicate requests, duplicate operations, broker restart, target restore, connection nonce reuse. |
| Validation | Unknown fields, extra parameters, malformed IDs, stale resources, oversized payloads. |
| Confirmation | Target-derived resource, mismatched controller resource, destructive confirmation omission. |
| Concurrency | Local versus remote mutation, two remote requests, per-target and fleet-wide limits. |
| Failure | DNS failure, timeout before acceptance, timeout after acceptance, link crash, broker crash, audit failure. |
| Audit | Issuer-qualified actor, operation correlation, no secrets, stable errors, unknown recovery. |

| Compatibility | Minor-version intersection, capability revision mismatch, major-version refusal. |
| UI | No literal templ calls, stale timestamps, disabled capabilities, no-JavaScript flows, mobile layout. |

## Observability

Expose bounded metrics and structured logs for:

- Pairing attempts, approvals, rejections, expiry, and rate limits.
- Active, stale, offline, revoked, incompatible, and conflicting peers.
- Request count and latency by fixed capability and stable outcome.
- Signature, pin, grant, replay, and version rejection counts.
- Operation state transitions and unknown outcomes.
- Inventory age and polling duration.

Never log invitation secrets, browser tokens, private keys, raw actor credentials, full signed payloads, arbitrary parameters, or backend error text. Machine IDs and operation IDs may be logged where needed for correlation; display names are supplemental.

## Rollout strategy

1. Ship identity and link packaging disabled by default.
2. Enable pairing only for development systems on a private test network.
3. Run read-only fleet MVP through at least one mixed-version upgrade cycle.
4. Validate Tailscale and plain routed-LAN deployments.
5. Release read-only fleet management before merging remote mutations.
6. Feature-gate the first mutation and require an explicit target grant.
7. Expand action coverage only after operation and audit data show stable behavior.

## Suggested pull request sequence

Keep reviews narrow. A reasonable sequence is:

1. Protocol ADR, threat model, and test vectors.
2. Strict query definitions and actor/audit schema.
3. Fleet database, operation ledger, and grant model.
4. Machine identity lifecycle.
5. Fleet socket and link process skeleton.
6. Pairing invitation and approval state machine.
7. TLS pinning, signed hello, and version negotiation.
8. Read-only inventory transport.
9. Broker-backed fleet list and target detail UI.
10. On-demand read-only queries.
11. First remote service-restart operation.
12. Key rotation, revocation, recovery, and hardening.

Every pull request must run `make build`, `make test`, `make fmt`, and `make lint`, or the matching `docker-*` targets. Protocol, broker, link, and parsing changes also require focused security tests and fuzzing before handoff.

## Open decisions before Phase 0 exits

- Exact initial fleet size and polling service-level objective.
- Exact initial capability bundle and first allowlisted restart units.
- JSON raw-payload signing versus a deterministic binary encoding for protocol version 1.
- TLS certificate validity and routine rotation intervals.
- Transport-key overlap duration and re-pairing procedure when the old key is unavailable.
- Grant default expiration and renewal policy.
- Permitted clock skew.
- Whether target invitation generation itself is sufficient local intent or final target approval is always mandatory. This plan recommends final approval.
- Whether read-only fleet access remains administrator-only in version 1. This plan recommends administrator-only access.
- Whether restored fleet identity is ever supported. This plan recommends reset and re-pairing for restored or cloned identity state.
